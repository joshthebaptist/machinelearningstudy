Part A – Fundamentals

We begin by understanding why linear models hit a wall, then build the intuition behind decision trees from the ground up.

Why Linear Models Fail

Analogy — Painting with a Ruler

Imagine you are given a ruler and asked to trace the outline of a coffee cup. You can approximate straight edges well, but the curved handle is impossible to capture accurately. A linear model is that ruler — it draws straight decision boundaries. When the true boundary is curved, no amount of cleverness with the ruler will produce an accurate result.

Linear classifiers (logistic regression, linear SVM) learn a boundary of the form:

w₁x₁ + w₂x₂ + … + wₙxₙ + b = 0

This is a hyperplane — a flat surface that separates the feature space into exactly two half-spaces. It works beautifully when the classes are linearly separable. But what happens when they are not?

The XOR Problem

The classic counter-example is the XOR (exclusive OR) function with two binary features:

x₁x₂Label
000
011
101
110

No single straight line can separate the 1s from the 0s. A linear model is forced to predict at most 50% accuracy — it has irreducible underfitting. A decision tree, however, can draw two axis-aligned cuts and perfectly separate the classes.

Curved Boundaries in Practice

Real-world data rarely falls into neat linear partitions. Consider:

  • Customer segmentation: High-income and low-income customers behave similarly (both buy luxury goods), while middle-income customers behave differently. The decision boundary curves around the middle.
  • Medical diagnosis: Both very low and very high blood pressure are dangerous — the risk surface is U-shaped, not linear.
  • Image classification: A pixel-intensity boundary between a cat and a dog is wildly non-linear.

Key Limitation

Linear models assume the decision boundary is a flat hyperplane. When the true boundary is curved, piecewise-linear, or involves feature interactions, linear models underfit. Decision trees solve this by recursively partitioning the feature space into rectangular regions, each with its own prediction.

What is a Decision Tree?

Definition

A decision tree is a supervised, non-parametric, tree-structured model that partitions the feature space into axis-aligned rectangular regions by recursively applying binary splits on individual features, and assigns a prediction (class label or numeric value) to each region.

Analogy — The 20 Questions Game

Think of a decision tree as a game of "20 Questions." You start with the entire dataset and ask one yes/no question at a time — each answer narrows the possibilities until you arrive at a final answer. "Is age > 40?" → Yes → "Is income > 70k?" → No → Predict: Low Risk. Each question is a split; each answer leads you down a branch; the final answer is a leaf.

More formally, a decision tree defines a piecewise-constant function over the feature space. Each leaf node corresponds to a rectangular region Rⱼ, and the prediction for any point in that region is the same — the majority class (classification) or the mean (regression).

Classification: ŷ = mode(y | x ∈ Rⱼ)

Regression: ŷ = (1/|Rⱼ|) Σ yᵢ for all i where xᵢ ∈ Rⱼ

Axis-Aligned Partitions

Every split in a standard decision tree tests a single feature against a threshold (e.g., "age ≤ 35?" or "income ≤ 50000?"). This means the decision boundaries are always parallel to the coordinate axes — horizontal or vertical lines in 2D, axis-aligned hyperplanes in higher dimensions.

Why Axis-Aligned?

Axis-aligned splits are simple, fast to evaluate, and interpretable. A single split partitions the current region into exactly two sub-regions. Oblique splits (e.g., "x₁ + x₂ ≤ 5") can reduce the number of splits needed but are harder to optimise and less interpretable. Standard implementations (scikit-learn, CART) use axis-aligned splits only.

Visual Intuition

Imagine a 2D scatter plot of two features. The first split draws a vertical line, dividing the plane into left and right. The second split draws a horizontal line inside one of those halves. Each subsequent split adds another line. After many splits, you have a patchwork of rectangles, each labelled with a class. This is the decision tree's partition of the feature space.

Anatomy of a Decision Tree

Every decision tree consists of three types of nodes connected by branches:

Node TypeRoleVisual Cue
Root Node The very first node at the top. Contains the entire dataset. Performs the first split. Top of the tree, no incoming branch
Internal Nodes Intermediate nodes that perform further splits on subsets of data. Has both incoming and outgoing branches
Leaf Nodes (Terminal) Endpoints that make predictions. No further splits occur here. Bottom of the tree, no outgoing branches
Branches Edges connecting nodes. Each branch represents the outcome of a test (e.g., Yes/No, Left/Right). Lines between nodes

Analogy — Flowchart

A decision tree is essentially a flowchart. The root is "Start here." Each internal node is a diamond-shaped decision box. Each branch is an arrow labeled "Yes" or "No." Each leaf is a rectangular outcome box. The path from root to leaf is one complete decision path.

Path from Root to Leaf

For any given data point, the prediction is determined by following the branches from root to leaf:

  1. Start at the root node (all data).
  2. At each internal node, evaluate the splitting condition (e.g., "age ≤ 35?").
  3. Follow the left branch if True, right branch if False.
  4. Repeat until you reach a leaf node.
  5. The leaf node's prediction is your output.

The depth of a tree is the maximum number of edges from the root to any leaf. The size is the total number of nodes (internal + leaves). A tree of depth d can have at most 2ᵈ leaf nodes.

Max leaves at depth d: L_max = 2ᵈ

Min depth to accommodate n leaves: d ≥ ⌈log₂(n)⌉

Key Properties of Decision Trees

1. Non-Parametric

Decision trees make no assumptions about the functional form of the relationship between features and target. Unlike linear regression (which assumes y = wx + b), a tree can learn arbitrarily complex mappings. The model's complexity grows with the data — more data can lead to deeper, more detailed trees.

Caveat

"Non-parametric" does not mean "no parameters." Trees have many structural parameters (depth, split conditions, leaf predictions). It means the number and nature of parameters are not fixed in advance.

2. Handles Mixed Feature Types

Decision trees natively handle:

  • Numerical features — split on thresholds (e.g., "income ≤ 50000")
  • Categorical features — split on subsets of categories (e.g., "color ∈ {red, blue}?")

Many other algorithms (SVM, linear regression) require all-numeric or one-hot-encoded inputs. Trees also handle missing values gracefully — some implementations route missing values down both branches with weighted contributions.

3. Naturally Multi-Class

Decision trees classify into any number of classes without modification. The splitting criteria (Gini, Entropy) generalise naturally: they compute impurity across all k classes, not just two. No need for one-vs-rest or one-vs-one strategies.

4. Feature Importance is Built-In

Because each split selects the feature that reduces impurity the most, the tree naturally ranks features by importance. A feature used at the root, with high impurity reduction, is more important than a feature used near the leaves with small reductions.

5. Interpretable

A trained decision tree can be visualised as a flowchart. Any stakeholder — doctor, loan officer, business manager — can trace the path from root to leaf and understand exactly why a prediction was made. This is a major advantage over black-box models.

Takeaway

Decision trees are powerful, flexible, interpretable models that make no distributional assumptions. Their main weakness — overfitting — is addressed by pruning and ensemble methods (Part B).

Step-by-Step Construction Algorithm

Decision trees are built top-down via a greedy, recursive partitioning algorithm (CART — Classification and Regression Trees). Here is the full procedure:

  1. Start with the entire dataset D at the root node.
  2. For each feature f and each possible threshold t:
    • Split D into left subset D_L = {x ∈ D : f(x) ≤ t} and right subset D_R = {x ∈ D : f(x) > t}.
    • Compute the impurity reduction (gain) of this split.
  3. Select the (feature, threshold) pair that maximises the impurity reduction.
  4. Apply this split, creating two child nodes.
  5. Recursively repeat steps 2–4 on each child node.
  6. Stop when a stopping criterion is met (see next section).
  7. Assign predictions to leaf nodes:
    • Classification: majority class in the leaf.
    • Regression: mean of target values in the leaf.

Computational Complexity

At each node, the algorithm must evaluate every feature at every unique threshold. For a dataset with n samples and p features:

Per node: O(p × n × log n)

Total tree building: O(p × n × log n × depth)

Sorting feature values reduces threshold evaluation to O(n) per feature after an initial O(n log n) sort. With early stopping and pruning, the effective depth is much smaller than n.

Greedy Nature

Greedy ≠ Global Optimum

The CART algorithm is greedy: it picks the best split at each node without considering the impact on future splits. The globally optimal tree (which minimises total impurity across all leaves) is an NP-hard problem. Greedy construction is fast and produces near-optimal trees in practice.

Stopping Criteria

If a tree is allowed to grow without restriction, it will keep splitting until every leaf is pure (all samples in the leaf belong to one class). This leads to severe overfitting. Stopping criteria halt tree growth early:

CriterionParameter (sklearn)Effect
Maximum depth max_depth Stops splitting when depth reaches the limit. Smaller values → simpler tree.
Minimum samples to split min_samples_split A node must have at least this many samples to be split further.
Minimum samples in leaf min_samples_leaf Every leaf must contain at least this many samples. Prevents tiny, overfit leaves.
Minimum impurity decrease min_impurity_decrease A split must reduce impurity by at least this amount to be accepted.
Maximum leaf nodes max_leaf_nodes Builds a best-first tree with at most this many leaves.
Maximum features max_features Number of features to consider per split (mainly for ensembles).

Takeaway

No single criterion is universally best. In practice, max_depth, min_samples_leaf, and min_samples_split are the most commonly tuned. Cross-validation is essential for selecting optimal values.

Worked Example — Loan Approval

Consider a bank deciding whether to approve loans based on three features:

  • Income (k$ per year): 25, 50, 75, 100, 35, 80, 45, 60, 90, 55
  • Credit Score: 600, 680, 720, 750, 620, 710, 650, 690, 740, 670
  • Employment (years): 1, 5, 8, 12, 2, 7, 3, 6, 10, 4

Target: Approved (1) or Denied (0)

Step 1 — Root Node (All 10 samples)

6 Approved, 4 Denied.

We evaluate every possible split for each feature:

Try Income ≤ 50k: Left: {25k(D), 35k(D), 45k(D)} → 0 Approved, 3 Denied. Right: {50k(A), 75k(A), 100k(A), 80k(A), 60k(A), 90k(A), 55k(A)} → 6 Approved, 0 Denied.

This split achieves perfect purity on both sides — Gini Gain is maximised. This is selected as the root split.

Step 2 — Right Child (Income > 50k)

All 7 samples are Approved. This node is pure (Gini = 0). It becomes a leaf node with prediction: Approve.

Step 3 — Left Child (Income ≤ 50k)

3 samples: all Denied. Pure leaf → prediction: Deny.

The resulting tree is extremely simple: just one split at the root! This is the greedy algorithm at work — it found the most informative feature first and got lucky with a perfect split.

Takeaway

In practice, perfect splits are rare. The algorithm picks the best available split at each step, even if it doesn't achieve perfect purity. Multiple levels of splitting are usually needed to reach acceptable impurity in all leaves.

Splitting Criteria

How does the algorithm choose which split is "best"? It uses an impurity measure — a function that quantifies how mixed the classes are in a node. The best split maximises the impurity reduction (gain) from parent to children.

The two most widely used criteria are Gini Impurity and Entropy. Both are derived from information theory and probability theory.

Gini Impurity

Definition

Gini Impurity measures the probability that a randomly chosen sample from a node would be misclassified if it were labelled according to the class distribution of that node. A Gini of 0 means the node is pure (all one class); a Gini of 0.5 (for binary classification) means maximum impurity (50/50 split).

Gini Impurity Formula

Gini(t) = 1 − Σᵢ₌₁ᵏ pᵢ²

where:

  • t = the node (subset of data)
  • k = number of classes
  • pᵢ = proportion of samples belonging to class i in node t

Range: 0 (pure) to 1 − 1/k (maximum impurity)

For binary classification (k=2): range is [0, 0.5]

Worked Example — Node A

Node A contains: 40 cats, 10 dogs (50 samples total)

p_cat = 40/50 = 0.8

p_dog = 10/50 = 0.2

Gini(A) = 1 − (0.8² + 0.2²)

Gini(A) = 1 − (0.64 + 0.04)

Gini(A) = 1 − 0.68

Gini(A) = 0.32

Worked Example — Node B

Node B contains: 25 cats, 25 dogs (50 samples total)

p_cat = 25/50 = 0.5

p_dog = 25/50 = 0.5

Gini(B) = 1 − (0.5² + 0.5²)

Gini(B) = 1 − (0.25 + 0.25)

Gini(B) = 1 − 0.50

Gini(B) = 0.50 (maximum impurity for binary)

Analogy — Coin Fairness

Gini Impurity is like measuring how "fair" a coin is. A loaded coin (80/20 heads/tails) has low impurity — you can predict the outcome most of the time. A perfectly fair coin (50/50) has maximum impurity — you're essentially guessing. Gini of 0 means the coin always lands the same way (pure node).

Why Squared Proportions?

The term pᵢ² is the probability that two randomly drawn samples (with replacement) from the node both belong to class i. The sum Σpᵢ² is the probability that both samples agree. Therefore 1 − Σpᵢ² is the probability they disagree — which is exactly the "impurity" or "confusion" in the node.

Takeaway

Gini Impurity is intuitive, fast to compute, and the default criterion in scikit-learn's DecisionTreeClassifier. It penalises distributions that are far from concentrated (pure) and rewards splits that create purer children.

Gini Gain (Impurity Reduction)

To evaluate a potential split, we compute the Gini Gain — how much impurity is reduced by splitting the parent into left and right children:

Gini Gain Formula

Gini_Gain = Gini(parent) − [ (N_L / N) × Gini(left) + (N_R / N) × Gini(right) ]

where:

  • N = total samples in parent
  • N_L = samples in left child
  • N_R = samples in right child
  • N_L + N_R = N

Analogy — Sorting Laundry

Imagine a pile of mixed laundry (shirts and pants). Before sorting, the "impurity" is high — you can't predict what you'll grab. After sorting into two baskets (shirts vs pants), each basket has low impurity. The Gini Gain measures how much "cleaner" the baskets are compared to the original pile. A good sort maximises this improvement.

Full Worked Calculation Example

Setup: 100 samples — 60 cats, 40 dogs at the parent node

Parent Gini:

p_cat = 0.6, p_dog = 0.4

Gini(parent) = 1 − (0.6² + 0.4²) = 1 − (0.36 + 0.16) = 1 − 0.52 = 0.48

Evaluate Split: "Weight ≤ 15 kg"

Left child (Weight ≤ 15 kg): 45 samples — 30 cats, 15 dogs

p_cat = 30/45 = 0.667, p_dog = 15/45 = 0.333

Gini(left) = 1 − (0.667² + 0.333²) = 1 − (0.444 + 0.111) = 1 − 0.555 = 0.444

Right child (Weight > 15 kg): 55 samples — 30 cats, 25 dogs

p_cat = 30/55 = 0.545, p_dog = 25/55 = 0.455

Gini(right) = 1 − (0.545² + 0.455²) = 1 − (0.297 + 0.207) = 1 − 0.504 = 0.496

Weighted Average Child Impurity:

W = (45/100) × 0.444 + (55/100) × 0.496

W = 0.45 × 0.444 + 0.55 × 0.496

W = 0.1998 + 0.2728 = 0.4726

Gini Gain:

Gini_Gain = 0.48 − 0.4726 = 0.0074

The algorithm would compare this gain against gains from all other possible splits (other features, other thresholds) and choose the one with the highest Gini Gain.

Worked Example — A Better Split

Alternative Split: "Ear Length ≤ 3 cm"

Left (Ear ≤ 3 cm): 40 samples — 38 cats, 2 dogs

p_cat = 0.95, p_dog = 0.05

Gini(left) = 1 − (0.9025 + 0.0025) = 1 − 0.905 = 0.095

Right (Ear > 3 cm): 60 samples — 22 cats, 38 dogs

p_cat = 0.367, p_dog = 0.633

Gini(right) = 1 − (0.134 + 0.401) = 1 − 0.535 = 0.465

Weighted Average:

W = (40/100) × 0.095 + (60/100) × 0.465 = 0.038 + 0.279 = 0.317

Gini Gain:

Gini_Gain = 0.48 − 0.317 = 0.163

This split has a gain of 0.163, much higher than the weight split (0.0074). The algorithm would choose the ear length split — it does a far better job of separating cats from dogs.

Entropy & Information Gain

Definition — Entropy

Entropy (Shannon Entropy) measures the uncertainty or disorder in a node. It originates from information theory and quantifies the average number of bits needed to encode the class label of a randomly chosen sample.

Entropy Formula

H(t) = − Σᵢ₌₁ᵏ pᵢ · log₂(pᵢ)

where:

  • t = the node
  • k = number of classes
  • pᵢ = proportion of class i in node t
  • Convention: 0 · log₂(0) = 0

Range: 0 (pure) to log₂(k) (maximum impurity)

For binary classification: [0, 1]

Worked Example — Entropy Calculation

Node with 70 positive, 30 negative (100 samples)

p_pos = 0.7, p_neg = 0.3

H(t) = −[0.7 · log₂(0.7) + 0.3 · log₂(0.3)]

log₂(0.7) = log(0.7)/log(2) = −0.1549/0.3010 = −0.5146

log₂(0.3) = log(0.3)/log(2) = −0.5229/0.3010 = −1.7370

H(t) = −[0.7 × (−0.5146) + 0.3 × (−1.7370)]

H(t) = −[−0.3602 + (−0.5211)]

H(t) = −[−0.8813]

H(t) = 0.8813 bits

Worked Example — Pure Node

Node with 100 positive, 0 negative

p_pos = 1.0, p_neg = 0.0

H(t) = −[1.0 · log₂(1.0) + 0.0 · log₂(0.0)]

H(t) = −[1.0 × 0 + 0] = 0

Pure nodes have zero entropy — no uncertainty at all.

Information Gain

Definition

Information Gain (IG) measures the reduction in entropy achieved by a split. It is the entropic analogue of Gini Gain.

Information Gain Formula

IG = H(parent) − Σⱼ (|childⱼ| / |parent|) × H(childⱼ)

Full IG Calculation

Parent: 60 positive, 40 negative (100 samples)

H(parent) = −[0.6 · log₂(0.6) + 0.4 · log₂(0.4)]

= −[0.6 × (−0.737) + 0.4 × (−1.322)]

= −[−0.442 + (−0.529)] = −[−0.971] = 0.971

After Split "Feature X ≤ 5":

Left: 50 samples — 45 pos, 5 neg

H(left) = −[0.9·log₂(0.9) + 0.1·log₂(0.1)] = −[0.9×(−0.152) + 0.1×(−3.322)] = 0.469

Right: 50 samples — 15 pos, 35 neg

H(right) = −[0.3·log₂(0.3) + 0.7·log₂(0.7)] = −[0.3×(−1.737) + 0.7×(−0.515)] = 0.881

Weighted Child Entropy:

W = (50/100) × 0.469 + (50/100) × 0.881 = 0.2345 + 0.4405 = 0.675

Information Gain:

IG = 0.971 − 0.675 = 0.296 bits

Takeaway

Information Gain is the criterion used by the ID3 and C4.5 algorithms. It favours splits that create the largest reduction in uncertainty. The split with the highest IG is selected at each node.

AspectGini ImpurityEntropy (Information Gain)

Analogy — Measuring Messiness

Gini and Entropy are like two different ways to measure how "messy" a room is. Gini counts the probability that two randomly picked items don't match. Entropy counts how many bits you'd need to describe the mess. Both agree on what's clean vs messy — they just measure it slightly differently. In practice, either measure gives you the same cleaning strategy.

When It Matters

For very large datasets or when class distributions are extremely skewed, Gini can be slightly faster. For small datasets with many classes, Entropy may produce slightly better-calibrated probability estimates. In practice, try both and pick the one that cross-validates better — but don't expect dramatic differences.

Overfitting & Pruning

Why Trees Overfit

An unconstrained decision tree will keep splitting until every leaf is pure — meaning each leaf contains only samples of one class. In effect, it memorises the training data, including noise and outliers. A tree with 10,000 leaves on 10,000 training samples has essentially built a lookup table — it will classify every training sample correctly but generalise terribly to new data.

Analogy — Studying for an Exam

An overfit tree is like a student who memorises every exact question and answer from past exams but doesn't understand the underlying concepts. When the real exam has slightly different wording, they fail. A well-pruned tree is like a student who learns the general principles — they can handle variations they haven't seen before.

There are two main strategies to combat overfitting: pre-pruning (stop early) and post-pruning (grow fully, then trim).

Pre-Pruning (Early Stopping)

Pre-pruning stops tree growth before it becomes too complex by setting constraints:

max_depth

Example: max_depth=3

The tree can have at most 3 levels of splits below the root. This produces at most 2³ = 8 leaf nodes. Each leaf represents a relatively large region of the feature space, reducing the risk of fitting noise.

Depth 0 (root): 1 node → 1 region

Depth 1: 2 nodes → 2 regions

Depth 2: 4 nodes → 4 regions

Depth 3: up to 8 nodes → up to 8 regions

min_samples_split

A node will only be split if it contains at least min_samples_split samples. Default is 2. Setting this to, say, 20 means nodes with fewer than 20 samples are never split, even if impure.

min_samples_leaf

Every leaf must contain at least min_samples_leaf samples. This prevents the creation of tiny leaves that capture individual data points (overfitting to outliers).

Rule of Thumb for min_samples_leaf

A common heuristic: min_samples_leaf ≈ 0.5% to 1% of total training samples.

For n=10,000: min_samples_leaf ∈ [50, 100]

Trade-Off

Pre-pruning is fast but can be too aggressive — it may stop splitting before the tree has captured important patterns (underfitting). This is called the horizon effect: a split that looks unpromising in isolation might lead to very informative splits deeper in the tree.

Post-Pruning

Post-pruning lets the tree grow fully, then removes branches that don't improve generalisation. The most principled method is Cost-Complexity Pruning (used by CART and scikit-learn).

Cost-Complexity Function

R_α(T) = R(T) + α × |T̃|

where:

  • R(T) = total impurity of tree T (sum of impurities in all leaves, weighted by sample count)
  • |T̃| = number of leaf nodes in T
  • α (ccp_alpha) = complexity parameter — controls the penalty for having more leaves

The pruning algorithm works as follows:

  1. Grow a full, unpruned tree T_max.
  2. For each internal node, compute the effective alpha at which it would be pruned:

α_eff = (R(t) − R(T_t)) / (|T̃_t| − 1)

where t is the node, T_t is the subtree rooted at t

  1. Prune the node with the smallest α_eff (least valuable subtree).
  2. Repeat until only the root remains.
  3. Use cross-validation to select the optimal α from the resulting sequence of trees.

Worked Example

Node t has impurity R(t) = 0.30 and its subtree has 4 leaves with total impurity R(T_t) = 0.20.

α_eff = (0.30 − 0.20) / (4 − 1) = 0.10 / 3 = 0.033

If ccp_alpha is set to 0.05, this node would be pruned because α_eff (0.033) < ccp_alpha (0.05).

Takeaway

Post-pruning is generally more reliable than pre-pruning because it lets the tree explore the full structure first, then decides what to keep. In scikit-learn, use ccp_alpha with cross-validation to find the optimal pruning level.

Bias-Variance Tradeoff in Trees

Tree TypeBiasVarianceWhen to Use
Shallow (depth 2–3) High — misses complex patterns Low — stable across datasets Small datasets, need interpretability, baseline model
Deep (unconstrained) Low — captures fine details High — varies wildly with training data Ensembles (random forests, gradient boosting) where variance is averaged out
Balanced (pruned/regularised) Moderate Moderate Single-tree deployment, interpretability + reasonable accuracy

Bias-Variance Decomposition (for squared error loss)

E[(y − f̂(x))²] = Bias²(f̂) + Var(f̂) + σ²

where σ² is irreducible noise.

Shallow tree: high Bias², low Var → total error dominated by bias

Deep tree: low Bias², high Var → total error dominated by variance

Goal: find the sweet spot that minimises total error.

Analogy — Camera Focus

A shallow tree is like a photo that's slightly blurry everywhere (high bias, low variance). A deep tree is like a photo that's razor-sharp on the subject but noisy in the background (low bias, high variance). The balanced tree is like a well-focused photo — sharp where it matters, clean elsewhere.

Feature Importance

Decision trees naturally rank features by how useful they are for prediction. There are two main approaches:

1. Impurity-Based Importance (Default in sklearn)

Formula

Importance(f) = Σ_{splits using f} (N_s / N) × ΔImpurity(s)

where:

  • N_s = number of samples at split s
  • N = total samples at root
  • ΔImpurity(s) = impurity reduction of split s

The sum is normalised so all features sum to 1.0.

Example

A tree uses "Income" at the root (100 samples, ΔGini = 0.30) and "Age" at depth 1 (40 samples, ΔGini = 0.10).

Income: (100/100) × 0.30 = 0.30

Age: (40/100) × 0.10 = 0.04

After normalisation across all features, Income might have importance 0.75 and Age 0.10 (with other features summing to 0.15).

2. Permutation Importance (More Reliable)

Randomly shuffle one feature's values and measure the drop in model accuracy. A large drop means the feature is important. This doesn't suffer from the bias towards high-cardinality features that impurity-based importance has.

Caveat

Impurity-based importance is biased towards features with many unique values (high cardinality). For example, a unique ID column would have high impurity-based importance but zero predictive value. Permutation importance is more trustworthy but computationally more expensive.

In scikit-learn

Accessing Feature Importances

tree.feature_importances_ — returns the normalised impurity-based importances as a numpy array.

forest.feature_importances_ — for a Random Forest, this averages importance across all trees.

To visualise: use plot_tree() or create a horizontal bar chart of importances.

Takeaway

Feature importance from trees is a quick, interpretable way to understand which features drive predictions. Use impurity-based importance for speed, permutation importance for reliability. Always check for correlated features, which can dilute importance scores.

Decision Trees for Regression

Decision trees can predict continuous values, not just classes. The key differences from classification:

AspectClassification TreeRegression Tree
Target Discrete class labels Continuous numerical values
Leaf prediction Majority class Mean of training samples in the leaf
Splitting criterion Gini Impurity / Entropy Variance Reduction (MSE)
Impurity measure Class proportions Variance (or MSE) of target values

Variance Reduction Criterion

Split quality is measured by the reduction in Mean Squared Error (MSE):

MSE(node) = (1/|node|) Σ (yᵢ − ȳ)²

where ȳ is the mean of the node.

Gain = MSE(parent) − [(N_L/N) × MSE(left) + (N_R/N) × MSE(right)]

Worked Example — House Prices

Parent node: [200k, 250k, 300k, 350k, 400k] → ȳ = 300k

MSE(parent) = [(200−300)² + (250−300)² + (300−300)² + (350−300)² + (400−300)²] / 5

= [10000 + 2500 + 0 + 2500 + 10000] / 5 = 25000 / 5 = 5000

After split "Sq Footage ≤ 1500":

Left: [200k, 250k] → ȳ = 225k, MSE = 625

Right: [300k, 350k, 400k] → ȳ = 350k, MSE = 1667

Weighted MSE = (2/5)×625 + (3/5)×1667 = 250 + 1000 = 1250

Gain = 5000 − 1250 = 3750

Step Function Output

No Extrapolation

A regression tree produces a step function — it predicts the same constant value for every sample in a leaf. The prediction is always the mean of training values in that leaf, which means:

  • It can never predict a value outside the range of the training data (no extrapolation).
  • The output is piecewise-constant, not smooth.
  • For extrapolation tasks (e.g., predicting future stock prices), regression trees alone are inadequate. Gradient boosting (Topic 5) addresses this.

Part B – Ensemble Methods

Single decision trees are powerful but unstable. Ensemble methods combine many trees to produce predictions that are more accurate and more stable than any individual tree.

Why Ensembles? The Problem with One Tree

The Instability Problem

Decision trees have high variance. If you change the training data slightly (add or remove a few samples), the resulting tree can be dramatically different — different splits, different structure, different predictions. This instability is the fundamental motivation for ensemble methods.

Analogy — Asking One Expert vs a Panel

If you ask one doctor for a diagnosis, you get one opinion — possibly brilliant, possibly wrong. If you ask 100 doctors and take a majority vote, you're much more likely to get the right answer, even if some individual doctors are wrong. Random Forests are the "panel of experts" approach to machine learning.

The mathematical foundation comes from the bias-variance decomposition:

If each tree has variance σ² and correlation ρ between trees, the ensemble variance is:

Var(ensemble) = ρσ² + (1−ρ)σ²/n

where n = number of trees.

As n → ∞: Var(ensemble) → ρσ²

To minimise ensemble variance, we need to minimise ρ (correlation between trees). Random Forests achieve this via bootstrap sampling AND feature randomisation.

Bootstrap Sampling (Bagging)

Bootstrap Sampling

A bootstrap sample is a random sample of size n drawn with replacement from the original dataset of size n. Each bootstrap sample is a different "view" of the data — some samples appear multiple times, others not at all.

The 63-2 Rule

Probability that a specific sample is NOT included in one bootstrap draw:

P(not selected once) = (1 − 1/n)ⁿ

As n → ∞: (1 − 1/n)ⁿ → 1/e ≈ 0.368 = 36.8%

So approximately 36.8% of samples are left out (Out-of-Bag samples) on average.

Conversely, approximately 63.2% of original samples appear at least once in the bootstrap sample.

Worked Example — n=10

P(a specific sample is not in one draw) = 9/10 = 0.9

P(not in any of 10 draws) = 0.9¹⁰ = 0.3487 ≈ 34.9%

Close to the theoretical 36.8% — the approximation improves with larger n.

Out-of-Bag (OOB) Samples

The ~37% of samples not used in a particular bootstrap are called Out-of-Bag (OOB) samples for that tree. They serve as a built-in validation set:

  • Each tree is tested on its OOB samples (never seen during training).
  • OOB predictions across all trees are aggregated → OOB score.
  • OOB score ≈ cross-validation score, but computed for free (no extra training needed).

OOB Score Computation

For each training sample xᵢ:

  1. Identify all trees where xᵢ was NOT in the bootstrap sample (its OOB trees).
  2. Get predictions from these OOB trees only.
  3. Aggregate (vote for classification, average for regression).
  4. Compare to the true label yᵢ.

OOB score = proportion of correctly classified samples across all xᵢ.

Bagging Algorithm

  1. Draw n_trees bootstrap samples from the training data.
  2. Train one decision tree on each bootstrap sample.
  3. For a new prediction:
    • Classification: majority vote across all trees.
    • Regression: average of all tree predictions.

Bagging alone reduces variance significantly, but trees trained on different bootstrap samples are still somewhat correlated (they see similar data). Feature randomisation further reduces this correlation.

Feature Randomisation

In a standard decision tree, every split considers all p features. In a Random Forest, each split considers only a random subset of m features. This decorrelates the trees — different trees focus on different features, making their errors more independent.

Default Feature Subset Sizes

Classification: m = √p (square root of total features)

Regression: m = p/3 (one-third of total features)

Worked Example — p=16 features

Classification: m = √16 = 4 features considered per split

Regression: m = 16/3 ≈ 5 features considered per split

At each node, the algorithm randomly selects 4 (or 5) features, then picks the best split among only those features.

max_features Parameter Options

ValueMeaning
'sqrt'√p features (default for classification)
'log2'log₂(p) features (more aggressive randomisation)
'None'All p features (equivalent to bagging, no randomisation)
Integer nExactly n features per split
Float f (0,1]f × p features per split (e.g., 0.3 means 30%)

Analogy — Committee with Diverse Expertise

Imagine assembling a committee to solve a problem. If every member has identical expertise (all p features), their opinions will be highly correlated — you don't gain much from the ensemble. But if each member specialises in a random subset of skills (m features), the committee is more diverse, and the collective wisdom is greater than any individual.

Takeaway

Feature randomisation is the key ingredient that makes Random Forests better than plain bagging. By restricting each tree to a random subset of features at each split, it forces the trees to be different from each other, which reduces ensemble variance and improves generalisation.

Random Forest Architecture

Definition

A Random Forest is an ensemble of decision trees where: (1) each tree is trained on a different bootstrap sample, and (2) at each split, only a random subset of features is considered. Predictions are aggregated by majority vote (classification) or averaging (regression).

Random Forest Architecture

Key properties:

  • Each tree is independent — can be trained in parallel.
  • Each tree sees a different subset of data AND features.
  • OOB samples for each tree provide free validation.
  • Ensemble variance decreases with more trees (diminishing returns).

Key Hyperparameters of Random Forests

ParameterDefault (sklearn)EffectRecommendation
n_estimators 100 Number of trees in the forest. More trees → better performance but slower. Diminishing returns after ~200-500. Start with 100, increase to 200-500 if needed.
max_depth None (unlimited) Maximum depth of each tree. Deeper trees → more complex, more overfitting risk. For large datasets: None is fine (ensemble handles variance). For small datasets: 10-20.
max_features 'sqrt' (classification) / 1.0 (regression) Features per split. Fewer → more diverse trees, less overfitting. More → trees more similar, less variance reduction. 'sqrt' for classification, 'None' or p/3 for regression.
min_samples_leaf 1 Minimum samples per leaf. Larger values → smoother predictions, less overfitting. 1 for large datasets. 5-20 for noisy/small datasets.
oob_score False If True, computes OOB score after training — free validation estimate. Set to True for quick model evaluation without cross-validation.
n_jobs None (1) Number of CPU cores for parallel training. -1 uses all cores. Set to -1 for faster training on multi-core machines.

Important Notes

  • n_estimators: Unlike most models, more trees virtually never hurt Random Forests (no overfitting from more trees), but training time increases linearly.
  • max_depth: In Random Forests, allowing deep trees is often fine because the averaging over many trees controls variance. But for very large forests (1000+ trees), you can increase depth for better accuracy.
  • max_features: This is the most important hyperparameter for controlling the bias-variance tradeoff in the ensemble. Lower values → more diversity → lower variance → but potentially higher bias.

Part C – Applications & Evaluation

Decision trees and Random Forests are applied across industries. In this section, we explore real-world use cases and the metrics used to evaluate them.

Application — Medical Diagnosis (Diabetes)

Predicting whether a patient has diabetes based on features like glucose level, BMI, age, blood pressure, and insulin.

Why Random Forests?

  • Interpretability: Feature importance tells doctors which measurements matter most (e.g., glucose level, BMI).
  • Handles mixed data: Numeric (glucose, age) and categorical (family history) features work seamlessly.
  • Robust to noise: Medical data often has measurement errors; ensemble averaging smooths these out.
  • Non-linear relationships: Diabetes risk has complex interactions (e.g., BMI matters more for certain age groups).

Evaluation Metric: Recall (Sensitivity)

Definition

Recall = TP / (TP + FN) — the proportion of actual positives correctly identified.

In diabetes diagnosis:

High recall = few missed diabetes cases (low false negatives).

This is critical because missing a diabetes case can lead to untreated disease, complications, and even death. A false alarm (false positive) is inconvenient but far less dangerous.

Example Calculation

A Random Forest classifier tested on 200 patients:

True Positives (diabetic, correctly flagged): 85

False Negatives (diabetic, missed): 15

False Positives (healthy, flagged as diabetic): 20

True Negatives (healthy, correctly cleared): 80

Recall = 85 / (85 + 15) = 85/100 = 0.85 (85%)

Precision = 85 / (85 + 20) = 85/105 = 0.810 (81%)

Application — Credit Risk Scoring

Banks use Random Forests to predict the probability of loan default. Features include income, credit history, employment status, debt-to-income ratio, and loan amount.

Fairness Considerations

Fairness in Credit Scoring

Models must not discriminate based on protected attributes (race, gender, age). Even if these features are excluded, correlated features (zip code, education) can encode bias. Auditing with fairness metrics (demographic parity, equalised odds) is essential.

Explainability with SHAP

SHAP (SHapley Additive exPlanations) provides per-patient explanations of the model's prediction. For each loan applicant, SHAP shows which features pushed the prediction toward "default" vs "no default" and by how much. This satisfies regulatory requirements (e.g., "right to explanation" under GDPR).

Evaluation Metric: Precision at Top Decile

Precision@Top Decile: Of the top 10% highest-risk applicants (as ranked by the model), what fraction actually defaulted?

This metric focuses on the most actionable segment — the bank will investigate or deny the top 10%. Higher precision here means fewer false alarms in the investigation queue.

Example

1000 applicants, 100 actual defaults.

Model ranks applicants by predicted default probability. Top 100 (top decile) are flagged.

Of these 100 flagged: 45 actually defaulted.

Precision@TopDecile = 45/100 = 0.45 (45%)

This means 45% of the flagged applicants are true defaults — the bank catches 45/100 = 45% of all defaults while only investigating 10% of applicants.

Application — Customer Churn Prediction

Predicting which customers will cancel their subscription. Features include usage frequency, contract length, monthly charges, tenure, and support calls.

Business-Aligned Metric Design

Why Accuracy is Misleading

If 95% of customers don't churn, a model that predicts "no churn" for everyone achieves 95% accuracy but catches zero churners. We need metrics aligned with the business goal.

Cost-Sensitive Evaluation

Business Cost Matrix:

False Negative (missed churner): Cost = $500 (lost lifetime value)

False Positive (false alarm): Cost = $50 (unnecessary retention offer)

Total Cost = FN × $500 + FP × $50

The optimal model minimises total cost, not total errors.

Model Comparison

Model A: 90% recall, 30% precision → catches 90 churners, but 210 false alarms. Cost = 10×500 + 210×50 = $5,000 + $10,500 = $15,500

Model B: 75% recall, 60% precision → catches 75 churners, 50 false alarms. Cost = 25×500 + 50×50 = $12,500 + $2,500 = $15,000

Model B is better despite lower recall — the cost savings from fewer false alarms outweigh the missed churners.

Takeaway

Always design evaluation metrics around the actual business objective. Accuracy, F1, and AUC are useful diagnostics, but the final model selection should be based on cost minimisation or revenue maximisation.

Gini Coefficient (for model evaluation)

Gini = 2 × AUC − 1

AUC of 0.5 (random) → Gini = 0 (no discrimination)

AUC of 1.0 (perfect) → Gini = 1.0 (perfect discrimination)

Industry standard in credit scoring: Gini > 0.4 is good, > 0.6 is excellent.

Choosing the Right Metric

  • Binary classification with balanced classes → Accuracy or F1
  • Imbalanced classes → F1, ROC-AUC, or Precision-Recall AUC
  • Cost-sensitive decisions → custom cost matrix
  • Ranking tasks (credit scoring) → Gini or AUC
  • Quick internal validation → OOB Error

Confusion Matrix — Foundation of All Metrics

Definition

A confusion matrix is a 2×2 table (for binary classification) that shows the counts of True Positives, False Positives, True Negatives, and False Negatives. All classification metrics are derived from these four numbers.

                        Predicted
                   ┌─────────┬─────────┐
                   │   Pos   │   Neg   │
        ┌──────────┼─────────┼─────────┤
  Actual│   Pos    │   TP    │   FN    │
        ├──────────┼─────────┼─────────┤
        │   Neg    │   FP    │   TN    │
        └──────────┴─────────┴─────────┘

  TP = True Positive:  predicted positive, actually positive
  FN = False Negative: predicted negative, actually positive (MISS!)
  FP = False Positive: predicted positive, actually negative (FALSE ALARM!)
  TN = True Negative:  predicted negative, actually negative

Worked Example — Diabetes Dataset

200 patients tested with a Random Forest classifier:

                        Predicted
                   ┌─────────┬─────────┐
                   │ Diabetes│ Healthy │
        ┌──────────┼─────────┼─────────┤
  Actual│ Diabetes │   85    │   15    │
        ├──────────┼─────────┼─────────┤
        │ Healthy  │   20    │   80    │
        └──────────┴─────────┴─────────┘

  TP = 85, FN = 15, FP = 20, TN = 80

Accuracy = (85+80)/(85+15+20+80) = 165/200 = 0.825

Precision = 85/(85+20) = 85/105 = 0.810

Recall = 85/(85+15) = 85/100 = 0.850

F1 = 2×(0.810×0.850)/(0.810+0.850) = 2×0.6885/1.660 = 0.830

Takeaway

The confusion matrix is the single source of truth for all classification evaluation. Always start here — compute the four counts, then derive whatever metrics you need. It reveals the type of errors the model makes, which is more informative than any single scalar metric.

Flashcards

What is the range of Gini Impurity for binary classification?

[0, 0.5] — 0 means pure (all one class), 0.5 means maximum impurity (50/50 split).

What is the formula for Gini Impurity?

Gini(t) = 1 − Σᵢ pᵢ²

where pᵢ is the proportion of class i in node t.

What percentage of samples are Out-of-Bag (OOB) in a bootstrap sample, approximately?

≈ 36.8% — derived from the limit (1 − 1/n)ⁿ → 1/e ≈ 0.368 as n → ∞.

Why do Random Forests use feature randomisation at each split?

To decorrelate the trees. If all trees used all features, they would be highly correlated and the ensemble variance reduction would be limited. Random subsets force trees to be diverse.

What is the default max_features for Random Forest classification in sklearn?

'sqrt' — the square root of the total number of features (p).

What is Information Gain?

IG = H(parent) − Σⱼ (|childⱼ|/|parent|) × H(childⱼ)

The reduction in entropy achieved by splitting a node. The split with the highest IG is selected.

What does cost-complexity pruning (ccp_alpha) do?

It penalises trees with more leaves. The cost function is R_α(T) = R(T) + α|T̃|. Higher α → more pruning → simpler tree. Cross-validation selects the optimal α.

What is the splitting criterion for regression trees?

Variance Reduction (MSE) — the split that maximises the reduction in Mean Squared Error of the target variable. Leaf predictions are the mean of the training samples in that leaf.

Can decision trees extrapolate (predict values outside the training range)?

No. Regression trees produce a step function where each leaf predicts the mean of its training samples. The prediction can never exceed the min or max of the training targets.

What is the Gini Coefficient used for in credit scoring?

Measuring ranking power: Gini = 2 × AUC − 1. It quantifies how well the model separates good borrowers from bad ones. A Gini of 0 means random, 1 means perfect separation.

What is the formula for the Entropy of a binary node with proportion p of the positive class?

H = −p·log₂(p) − (1−p)·log₂(1−p)

Range: [0, 1] for binary classification. Maximum at p=0.5.

What problem does Random Forest solve that a single decision tree cannot?

High variance (instability). A single tree's predictions change dramatically with small data changes. Averaging many decorrelated trees reduces variance while maintaining low bias.

Scenario Question

Scenario: Fraud Detection at a Payment Company

A payment processing company wants to detect fraudulent transactions. They have 10 million transactions per day, of which only 0.1% (10,000) are fraudulent. They plan to deploy a Random Forest model and need to evaluate its performance.

The team's initial model achieves 99.9% accuracy by predicting "not fraud" for every transaction. The CFO is delighted, but the data science team is concerned.

Questions:

  1. Why is 99.9% accuracy misleading in this scenario?
  2. What metrics should the team focus on instead, and why?
  3. The team retrains with a tuned Random Forest (max_features='sqrt', n_estimators=500, min_samples_leaf=10). The new confusion matrix shows: TP=8,500, FN=1,500, FP=50,000, TN=9,940,000. Calculate: Accuracy, Precision, Recall, F1.
  4. If each missed fraud costs $200 and each false alarm costs $5, what is the total daily cost of this model? Is it better or worse than the "predict all not fraud" baseline?
  5. What hyperparameter changes could reduce false alarms without sacrificing too much recall?

1. Why 99.9% Accuracy is Misleading

A "predict all not fraud" model achieves 99.9% accuracy because 99.9% of transactions are legitimate. But it catches zero frauds — its recall is 0%. The model is useless for its intended purpose. Accuracy is meaningless with extreme class imbalance.

2. Better Metrics

  • Recall: Of actual frauds, how many did we catch? Critical — missed frauds cost real money.
  • Precision: Of flagged transactions, how many are actually fraud? Important because investigations have costs.
  • F1 Score: Balances precision and recall.
  • Cost-based metric: Total cost = FN × $200 + FP × $5. This is the ultimate business metric.
  • Precision-Recall AUC: Better than ROC-AUC for highly imbalanced data.

3. Calculations

TP=8,500, FN=1,500, FP=50,000, TN=9,940,000, Total=10,000,000

Accuracy = (8,500 + 9,940,000) / 10,000,000 = 9,948,500 / 10,000,000 = 99.485%

Precision = 8,500 / (8,500 + 50,000) = 8,500 / 58,500 = 0.1453 (14.53%)

Recall = 8,500 / (8,500 + 1,500) = 8,500 / 10,000 = 0.85 (85%)

F1 = 2 × (0.1453 × 0.85) / (0.1453 + 0.85) = 2 × 0.1235 / 0.9953 = 0.2481

4. Cost Analysis

Tuned model: Cost = 1,500 × $200 + 50,000 × $5 = $300,000 + $250,000 = $550,000/day

"All not fraud" baseline: Cost = 10,000 × $200 + 0 × $5 = $2,000,000/day

The tuned model saves $1,450,000/day — a massive improvement despite the low precision. Even flagging many false alarms is worth it because each caught fraud saves $200 and each false alarm only costs $5.

5. Reducing False Alarms

  • Increase min_samples_leaf → smoother decision boundary, fewer edge-case flags.
  • Increase min_impurity_decrease → only split on highly informative features.
  • Increase max_depth slightly → deeper trees can learn more nuanced patterns, potentially reducing false positives.
  • Adjust the classification threshold → raise it to be more conservative about flagging fraud.
  • Use class_weight='balanced' to upweight the minority class without over-flagging.

Final Recap

Key Concepts Summary

  • Decision Trees partition feature space into axis-aligned rectangles via recursive greedy splitting. They are non-parametric, handle mixed features, and are naturally interpretable.
  • Splitting Criteria: Gini Impurity (1 − Σpᵢ², range [0, 0.5] for binary) and Entropy (−Σpᵢ log₂pᵢ, range [0, 1] for binary) measure node impurity. The best split maximises the impurity reduction (gain).
  • Overfitting is the main weakness of single trees. Combat it with pre-pruning (max_depth, min_samples_leaf) or post-pruning (cost-complexity pruning with ccp_alpha).
  • Feature Importance is computed from the sum of impurity reductions at each split using that feature, normalised across all features.
  • Regression Trees use variance (MSE) reduction and predict the leaf mean. They produce step functions and cannot extrapolate.
  • Random Forests ensemble many decorrelated trees via bootstrap sampling (Bagging) and feature randomisation (m = √p for classification, p/3 for regression).
  • OOB Samples (~37% of data) provide free validation — OOB score ≈ cross-validation performance.
  • Evaluation requires choosing metrics aligned with the business objective: recall for medical diagnosis, precision@top decile for credit scoring, cost matrices for churn prediction.
  • The Confusion Matrix (TP, FP, TN, FN) is the foundation from which all classification metrics derive.

Big Picture Takeaway

Decision trees provide the building blocks; Random Forests provide the engineering that makes them production-ready. Understanding the bias-variance tradeoff, splitting criteria, and ensemble mechanics gives you the foundation to build, tune, and evaluate tree-based models for any classification or regression task.
