The Generalisation Problem

The entire purpose of machine learning is not to memorise the data we already have, but to make accurate predictions on data we have never seen before. This gap between "doing well on training data" and "doing well in the real world" is the central challenge of model validation.

Definition — Generalisation

Generalisation is a model's ability to perform well on unseen data drawn from the same underlying distribution as the training data. A model that generalises has learned the true signal rather than the noise.

Good Model vs Overfit Model

Consider a student preparing for an exam. One student studies past papers and understands the underlying concepts — when new questions appear on the exam, they can reason through them. Another student memorises the exact answers to past papers word-for-word — when the exam rephrases the questions, they freeze. The first student has generalised; the second has overfit.

✓ Good Model

  • Learns the true signal (the pattern underlying the data)
  • Training error is low but not zero
  • Test error is approximately equal to training error
  • Performs well on unseen, real-world data
  • Has appropriately matched complexity to the data

✗ Overfit Model

  • Memorises noise and idiosyncrasies of the training set
  • Training error tends toward zero
  • Test error is very high (diverges)
  • Fails catastrophically on new examples
  • Has excessive complexity relative to data availability

Analogy — The GPS Navigator

Imagine training a GPS system. A good model learns the general rules of navigation: major roads connect cities, highways have exits, rivers require bridges. An overfit model memorises every single turn you made during training — it knows you turned left at the coffee shop on Main Street, but when it encounters a road it hasn't seen, it has no idea where to go. The overfit GPS is brilliant on routes it has memorised and useless everywhere else.

Why Validation Reveals the Gap Before Deployment

If you only measure training performance, every model looks fantastic — even a model that has simply memorised the data. Validation provides an unbiased estimate of how the model will perform in production by evaluating it on data it has never trained on.

Common Mistake — Reporting Training Accuracy

Students often train a model, compute accuracy on the training set, and report it as the model's performance. This is meaningless — a decision tree with unlimited depth achieves 100% training accuracy every time. Always evaluate on held-out data.

Key Takeaway

A model is only as good as its performance on data it has never seen. Validation is the mechanism that tells us whether we have learned the signal or merely memorised the noise — before we deploy and face real consequences.

The Three-Way Split

To properly estimate generalisation, we split our data into three distinct sets. Each serves a specific, non-interchangeable purpose.

Definition — Train / Validation / Test Split

The training set is used to fit model parameters. The validation set is used for model selection and hyperparameter tuning. The test set is reserved for the final, unbiased evaluation of the chosen model's generalisation performance.

Roles of Each Set

Set Purpose Rule
Training Set (~60-70%) Fit the model's internal parameters (weights, coefficients, split points) Used during .fit()
Validation Set (~15-20%) Select the best model, tune hyperparameters, compare algorithms Used during model selection — NOT for training
Test Set (~15-20%) Final, unbiased estimate of generalisation performance NEVER touched until the very end

Critical Rule — The Test Set Is Sacred

The test set must never be used for any decision — not for hyperparameter tuning, not for feature selection, not for early stopping. If you peek at test performance and then adjust your model, the test estimate becomes optimistically biased. It is no longer an honest measure of generalisation. Think of it as a sealed envelope you only open on presentation day.

Why Not Just Train/Test?

If we only split into train and test, we would have to use the test set for hyperparameter tuning. This means the test set has influenced our model choice, so the test performance is no longer an unbiased estimate of real-world performance. The validation set acts as a "practice exam" — you can study from it and improve — while the test set is the actual exam.

Analogy — Dress Rehearsal

A theatre company has three stages of preparation: rehearsals (training set — learning lines and choreography), dress rehearsal (validation set — full run-through with costumes and lights, where feedback is gathered and adjustments made), and opening night (test set — the final performance where the audience sees the result and no changes are possible).

K-Fold CV Replaces Fixed Validation with Limited Data

When data is scarce, setting aside 20% for validation wastes precious training examples. K-Fold Cross-Validation solves this by rotating which portion serves as validation, so every sample gets a chance to be both training and validation data. We explore this in depth in Section 4.

Key Takeaway

The three-way split separates the roles of data: training for learning, validation for selecting, and testing for final honest evaluation. Violating these boundaries — especially using the test set for decisions — produces unreliable performance estimates.

The Bias-Variance Trade-Off

Every model's expected test error can be decomposed into three fundamental components. Understanding this decomposition is essential for diagnosing whether a model underfits, overfits, or is well-calibrated.

Decomposition of Expected Test Error E[Error] = Bias² + Variance + Irreducible Noise (σ²)

Breaking Down Each Component

Bias (Systematic Error)

Bias is the error introduced by approximating a complex real-world relationship with a simplified model. High bias means the model makes strong assumptions that prevent it from capturing the true pattern. It consistently misses the target — like an archer whose bow is sighted incorrectly: every arrow lands in the same wrong spot, but tightly clustered.

Variance (Sensitivity to Data)

Variance is the error introduced by the model's sensitivity to fluctuations in the training data. High variance means the model learns different patterns depending on which specific training samples it sees. Like an archer with an unsteady hand: the arrows scatter widely around the target, sometimes hitting the bullseye by luck, sometimes missing badly.

Irreducible Noise (σ²)

Irreducible noise is the inherent randomness in the data that no model can eliminate — measurement error, missing variables, stochastic processes. Even a perfect model with infinite data would still have this error. It sets the theoretical floor for model performance.

Full Expansion — Squared Error Decomposition E[(y − f̂(x))²] = [Bias(f̂(x))]² + Var(f̂(x)) + σ²

where:
Bias(f̂(x)) = E[f̂(x)] − f(x) (average prediction minus true value)
Var(f̂(x)) = E[(f̂(x) − E[f̂(x)])²] (variance of predictions across training sets)

The Three Regions

Region Bias Variance Behaviour
Underfitting HIGH Low Model is too simple. Cannot capture the underlying pattern. Both training and test error are high.
Sweet Spot Low Low Model complexity matches the data. Bias² + Variance is minimised. Best generalisation.
Overfitting Low HIGH Model is too complex. Fits training noise. Training error near zero, test error explodes.

Analogy — Throwing Darts at a Board

Imagine you are throwing darts at a bullseye, and you throw many rounds, each time with a slightly different dartboard (representing different training sets):

  • High Bias, Low Variance (Underfitting): All your darts land tightly clustered together, but they are centred in the top-left corner — far from the bullseye. You are consistently wrong, but consistently wrong in the same way. A model that always predicts the mean of the training data has this property.
  • Low Bias, High Variance (Overfitting): Your darts are scattered all over the board. Some land near the bullseye, some in the corners. The average position is roughly the centre, but each individual throw is unreliable. A high-degree polynomial that fits every training point perfectly has this property.
  • Low Bias, Low Variance (Sweet Spot): Your darts cluster tightly around the bullseye. Each throw lands close to the centre, and over many throws they stay there. This is the goal — a model that consistently predicts close to the true value.

Visualising the Trade-Off

As model complexity increases from left to right:

  • Bias² decreases monotonically — more complex models fit the training data better
  • Variance increases monotonically — more complex models are more sensitive to which training samples they see
  • Total Error follows a U-shape — it initially decreases (bias drops faster than variance rises), reaches a minimum, then increases (variance dominates)
  • The optimal model sits at the bottom of the U

Recap — Bias-Variance Trade-Off

  • Expected Test Error = Bias² + Variance + Noise
  • Increasing model complexity reduces bias but increases variance
  • The goal is to find the complexity level where total error is minimised
  • Underfitting: too simple → high bias → poor on both train and test
  • Overfitting: too complex → high variance → great on train, terrible on test
  • Validation (and CV) helps us find the sweet spot without touching the test set

K-Fold Cross-Validation

When data is limited, holding out a fixed validation set wastes valuable training examples. K-Fold Cross-Validation (CV) solves this by rotating which portion of data serves as validation, ensuring every sample is used for both training and evaluation.

Step-by-Step Derivation

Worked Example — K-Fold CV Algorithm
1 Partition
Randomly shuffle the dataset and divide it into K equal (or nearly equal) folds.
Example: K = 5 folds, N = 1000 samples → each fold has 200 samples.
2 Iterate
For i = 1 to K:
  • Hold out fold i as the validation set
  • Train the model on the remaining K − 1 folds (the training set)
  • Evaluate the trained model on fold i, recording the score
3 Average
CV Score = (1/K) × Σ Score_i
Standard Deviation = sqrt((1/K) × Σ (Score_i − CV Score)²)
The mean gives the performance estimate; the std dev gives the variability.

Worked Example — 3-Fold CV for House Price Prediction

Worked Example — 3-Fold CV with 9 Samples
1 Setup
We have 9 house samples. We want to predict price using a regression model.
Evaluation metric: RMSE (Root Mean Squared Error) in thousands of dollars.
2 Partition into 3 Folds
Fold 1: Samples {1, 2, 3}
Fold 2: Samples {4, 5, 6}
Fold 3: Samples {7, 8, 9}
3 Run 3 Training Rounds
Round 1: Train on {4,5,6,7,8,9}, evaluate on {1,2,3} → RMSE₁ = 12.4k
Round 2: Train on {1,2,3,7,8,9}, evaluate on {4,5,6} → RMSE₂ = 14.1k
Round 3: Train on {1,2,3,4,5,6}, evaluate on {7,8,9} → RMSE₃ = 11.8k
4 Compute CV Score
CV Score = (12.4 + 14.1 + 11.8) / 3 = 38.3 / 3 = 12.77k

Variance = [(12.4 − 12.77)² + (14.1 − 12.77)² + (11.8 − 12.77)²] / 3
  = [0.1369 + 1.7689 + 0.9409] / 3
  = 2.8467 / 3
  = 0.9489

Std Dev = √0.9489 = 0.96k

Final: 12.77k ± 0.96k
5 Interpretation
We expect our model to make predictions with an average RMSE of approximately $12,770 ± $960. The relatively small standard deviation suggests the model's performance is stable across different data partitions.

Why K-Fold CV is Better Than a Single Split

  • Every sample is used for both training and validation — no data is wasted
  • The performance estimate is more reliable — it is averaged over K different train/test combinations
  • The variance of the estimate tells us how stable the model is — high variance across folds signals that performance is data-dependent
  • Reduces the luck factor — a single validation split might be unusually easy or hard by chance

Choosing K

K Value When to Use Trade-Off
K = 5 Standard choice for most datasets Good balance of bias and computational cost
K = 10 Larger datasets, or when more reliable estimates needed Lower bias but 2× computational cost vs K=5
K = N (LOO) Very small datasets (N < 50) Lowest bias, but highest computational cost and highest variance
K = 3 Very large datasets where speed matters Higher bias, but fast

Warning — Leave-One-Out (LOO) CV

LOO is computationally expensive (N model fits) and has high variance because each training set differs by only one sample. The K training sets overlap heavily, making the estimates highly correlated. Use LOO only when data is extremely scarce and you need every possible training sample.

Key Takeaway

K-Fold CV is the gold standard for estimating model performance with limited data. Choose K = 5 or K = 10 as default; use LOO for tiny datasets. Always report both the mean and the standard deviation across folds.

CV Strategies — When to Use What

Different data distributions and problem types require different cross-validation strategies. Using the wrong strategy can lead to data leakage or unreliable estimates.

Strategy How It Works When to Use Pros Cons
Standard K-Fold Randomly split data into K folds; rotate validation fold General-purpose regression and classification with IID data Simple, reliable, widely understood Not suitable for time series or grouped data
Stratified K-Fold Each fold preserves the class distribution of the full dataset Classification with imbalanced classes Ensures every fold represents all classes proportionally Slightly more complex to implement; not for regression
Leave-One-Out (LOO) K = N; each sample is its own fold Very small datasets (N < 50) Maximum training data per fold; no randomness in splits N model fits; high variance; computationally expensive
Time-Series Split Training set always precedes validation set in time order Sequential/temporal data (stocks, weather, logs) Respects temporal ordering; no future data leakage Later folds have much more training data; earlier folds may be too small
Group K-Fold All samples from the same group stay in the same fold Data with groups (patients, users, geographic regions) Prevents leakage between related samples Fewer effective training samples; more variance in estimates

Analogy — Exam Proctoring Strategies

  • Standard K-Fold is like randomly assigning students to study groups — anyone can be with anyone.
  • Stratified K-Fold ensures each study group has a proportional mix of strong, average, and weak students — nobody gets a group of only top performers.
  • Time-Series Split means you can only study with people from the past semester — you never use information from the future to prepare.
  • Group K-Fold keeps siblings in the same study group — because if one sibling shares their notes, the other knows the content too.

Data Leakage — The Silent Killer

Data leakage occurs when information from outside the training set leaks into the model. Using standard K-Fold on time-series data is the classic mistake — a model trained on days 1–50 validated on day 60 might have features computed from day 55, which is before the validation date but after the training cutoff. Always use Time-Series Split for temporal data.

Hyperparameter Optimisation

Parameters vs Hyperparameters

Definition — Parameters vs Hyperparameters

Parameters are learned from data during training (e.g., weights in a neural network, coefficients in linear regression, split thresholds in a decision tree). Hyperparameters are set before training and control the learning process (e.g., learning rate, regularisation strength λ, max depth of a tree, K in K-Nearest Neighbours).

The Hyperparameter Search Problem θ* = argmin_θ CV_Score(θ)

where θ = set of hyperparameters, CV_Score(θ) = K-Fold CV score for configuration θ

Search Strategies

1. Grid Search (Exhaustive Search)

How Grid Search Works
1 Define the Search Space
learning_rate ∈ {0.001, 0.01, 0.1, 1.0}
max_depth ∈ {3, 5, 7, 10}
n_estimators ∈ {100, 200, 500}
2 Enumerate All Combinations
Total combinations = 4 × 4 × 3 = 48 configurations
With 5-Fold CV: 48 × 5 = 240 model fits
3 Select the Best
Choose the θ with the lowest mean CV score.

Pros: Guarantees finding the best combination within the grid. Simple to implement.

Cons: Computational cost grows exponentially with the number of hyperparameters (curse of dimensionality). Wastes time evaluating unimportant parameter combinations.

2. Random Search

Instead of exhaustively searching every combination, Random Search samples a fixed number of random configurations from the search space.

How Random Search Works
1 Define Distributions
learning_rate ~ LogUniform(0.0001, 1.0)
max_depth ~ UniformInt(2, 20)
n_estimators ~ UniformInt(50, 1000)
2 Sample N Configurations
With n_iter=60: 60 random configurations × 5-fold CV = 300 model fits
But we explore a much wider range of values than Grid Search
3 Select the Best
Same as Grid Search — pick the configuration with lowest CV score.

Pros: Faster than Grid Search; explores more of the search space; works well when some hyperparameters matter more than others.

Cons: No guarantee of finding the absolute best; may miss narrow optima. Still requires specifying the number of iterations.

3. Bayesian Optimisation

Bayesian Optimisation builds a probabilistic model of the objective function (CV score as a function of hyperparameters) and uses it to decide which configuration to try next. It balances exploration (trying uncertain regions) and exploitation (refining promising regions).

Analogy — Finding the Best Restaurant

  • Grid Search is visiting every restaurant in the city in a systematic order. Exhaustive but takes forever.
  • Random Search is picking restaurants at random from the phone book. Faster, covers more ground, but no memory of past meals.
  • Bayesian Search is visiting restaurants, remembering which ones were good, and deliberately trying new places in the neighbourhoods that seem promising while occasionally exploring unfamiliar areas.

Pros: More efficient — finds good configurations with fewer evaluations; adapts based on past results.

Cons: More complex to implement; parallelisation is harder since each new evaluation depends on previous results.

Nested Cross-Validation

Definition — Nested CV

Nested CV uses an outer loop for unbiased performance estimation and an inner loop for hyperparameter selection. The outer loop estimates generalisation; the inner loop tunes hyperparameters within each outer fold.

Worked Example — Nested CV Structure
1 Outer Loop (5-Fold)
Split data into 5 outer folds. For each outer fold i:
  • Outer train = 4 folds, Outer test = 1 fold (hold out)
2 Inner Loop (3-Fold)
On the outer train set, run 3-Fold CV to find the best hyperparameters.
This is where hyperparameter tuning happens.
3 Evaluate
Retrain with the best hyperparameters on the full outer train set.
Evaluate on the outer test fold → record score.
4 Final Score
Average the 5 outer fold scores → unbiased estimate of generalisation.

Common Mistake — Confusing Inner and Outer CV Scores

The inner CV score tells you which hyperparameters work best on the training data. The outer CV score tells you how the model with those hyperparameters generalises to unseen data. Never report the inner CV score as your model's performance — it is optimistically biased because the hyperparameters were chosen to minimise it. Always report the outer CV score.

Nested CV — Total Model Fits Total Fits = Outer_K × Inner_K × n_configs
Example: Outer=5, Inner=3, 50 configs = 5 × 3 × 50 = 750 fits

Key Takeaway

Hyperparameter tuning is itself a form of learning — it must be contained within the cross-validation pipeline to avoid overfitting. Nested CV separates the "tuning" loop from the "evaluation" loop, giving you an honest performance estimate. Use Random Search as default; switch to Bayesian for expensive evaluations.

ERM Re-Anchor

We revisit the Empirical Risk Minimisation (ERM) framework from Topic 1 and show how each of its fundamental problems connects to the validation and generalisation tools we have just learned.

Definition — Empirical Risk Minimisation (ERM)

ERM is the principle of choosing the hypothesis h ∈ H that minimises the average loss over the training data:

h* = argminh ∈ H (1/n) Σᵢ L(h(xᵢ), yᵢ)

The empirical risk R̂(h) approximates the true risk R(h) = E[L(h(x), y)].

Three Problems of ERM and Their Solutions

Problem Description Solution
Problem 1: ERM overfits Minimising empirical risk alone leads to memorisation — the model achieves zero training error but fails to generalise. More complex models always fit training data better, so ERM without constraint always prefers the most complex hypothesis. K-Fold Cross-Validation — provides an unbiased estimate of true risk, allowing us to select the model that generalises rather than the one that memorises.
Problem 2: Unknown true risk minimiser Even if we know the best hypothesis exists, we cannot access the true data distribution to find it. We only have finite samples, which may not represent the full distribution. Regularisation — constrains the hypothesis space, reducing the gap between empirical and true risk by limiting model complexity (e.g., L1, L2 penalties).
Problem 3: Which hypothesis class? ERM takes the hypothesis class H as given. But how do we choose H? A linear model, a decision tree, and a neural network have very different H. Hyperparameter Optimisation — systematically searches over different hypothesis classes (via their hyperparameters) using CV to find the best one.
The ERM Framework with Solutions True Risk: R(h) = E[L(h(x), y)] ← Unknown, cannot compute
Empirical Risk: R̂(h) = (1/n) Σ L(h(xᵢ), yᵢ) ← What ERM minimises
Regularised Risk: R̂(h) + λΩ(h) ← Constrains complexity
CV Estimate: (1/K) Σ Score_k ← Unbiased estimate of R(h)

Analogy — Studying for an Unknown Exam

You are preparing for an exam but you don't know the exact questions (true risk is unknown). You have a set of past papers (training data). ERM says: find the study strategy that scores highest on these past papers. But this might mean memorising exact answers. Cross-validation is like having a friend hold back some past papers for a mock exam — it tests whether your strategy actually works. Regularisation is like forcing yourself to understand concepts rather than memorise — it constrains your approach to be simpler and more robust. Hyperparameter optimisation is choosing between studying techniques (flashcards vs practice problems vs summary notes).

Key Takeaway

The ERM framework is the theoretical foundation; CV, regularisation, and hyperparameter optimisation are the practical tools that address its three fundamental weaknesses. Together, they form the complete toolkit for building models that generalise.

Regularisation

Regularisation adds a penalty for model complexity to the loss function, discouraging the model from fitting noise. It is the mathematical embodiment of Occam's Razor: among models with similar performance, prefer the simpler one.

Regularised Loss Function L_regularised = L_original + λ × Ω(w)

where:
L_original = original loss (e.g., MSE, cross-entropy)
λ (lambda) = regularisation strength (hyperparameter)
Ω(w) = penalty on the model parameters

L2 Regularisation (Ridge)

Definition — L2 / Ridge Regularisation

L2 Regularisation adds the sum of squared weights to the loss function. It shrinks all weights toward zero but rarely makes them exactly zero. This prevents any single feature from dominating the prediction.

L2 (Ridge) Loss L_Ridge = L_original + λ × Σᵢ wᵢ²
λ > 0: penalises large weights. λ = 0: no regularisation. λ → ∞: all weights → 0.
Worked Example — L2 Regularisation Effect
1 Without Regularisation
Original Loss = 2.5, Weights: w₁=3.0, w₂=−2.5, w₃=1.8
Total Loss = 2.5 + 0 = 2.5
2 With λ = 0.1 (Light Regularisation)
Penalty = 0.1 × (3.0² + 2.5² + 1.8²) = 0.1 × (9.0 + 6.25 + 3.24) = 0.1 × 18.49 = 1.849
Total Loss = 2.5 + 1.849 = 4.349
Model must now balance fitting data (loss=2.5) against keeping weights small (penalty=1.849)
3 Effect on Weights
The optimiser will find weights that slightly increase training loss but are smaller in magnitude:
New weights might be: w₁=2.1, w₂=−1.8, w₃=1.3 (all shrunk toward zero)
New Loss: 2.9 (slightly worse) but Penalty: 0.1 × (4.41+3.24+1.69) = 0.934
Total: 2.9 + 0.934 = 3.834 → better than 4.349

L1 Regularisation (Lasso)

Definition — L1 / Lasso Regularisation

L1 Regularisation adds the sum of absolute weights to the loss function. Unlike L2, it can drive weights to exactly zero, effectively performing automatic feature selection.

L1 (Lasso) Loss L_Lasso = L_original + λ × Σᵢ |wᵢ|
λ > 0: penalises large weights. Drives irrelevant feature weights to exactly 0.

L1 vs L2 Comparison

Property L2 (Ridge) L1 (Lasso)
Penalty λ × Σwᵢ² λ × Σ|wᵢ|
Effect on small weights Shrinks toward zero Can set to exactly zero
Feature selection No — retains all features Yes — zeroes out irrelevant features
When to use All features potentially relevant Many features, suspect some are irrelevant
Geometry Circular constraint region (smooth) Diamond constraint region (corners on axes)

Analogy — Packing for a Trip

L2 (Ridge) is like a strict weight limit on your suitcase — you must reduce the weight of every item proportionally. Nothing gets left behind, but everything gets lighter. L1 (Lasso) is like a hard item limit — you can only bring 3 things. Some items get completely removed, while the ones you keep stay at full strength. The "diamond" shape of the L1 constraint means the optimal solution often lands on an axis (weight = 0).

The Role of λ

  • λ = 0: No regularisation — pure ERM (risk of overfitting)
  • λ small: Light regularisation — slight shrinkage, mild protection against overfitting
  • λ moderate: Balanced — the sweet spot found via cross-validation
  • λ large: Heavy regularisation — all weights near zero (risk of underfitting)

The optimal λ is chosen via cross-validation, treating it as a hyperparameter. This connects directly back to the ERM framework: regularisation addresses the problem of unknown true risk minimisers by constraining the hypothesis space.

Key Takeaway

L2 (Ridge) shrinks all weights — use when all features matter. L1 (Lasso) drives weights to zero — use for feature selection. λ controls the bias-variance trade-off: higher λ increases bias but decreases variance. Always tune λ via cross-validation.

Preventing Overfitting

Overfitting is the most common failure mode in machine learning. Fortunately, there are multiple well-established techniques to combat it, each addressing different aspects of the bias-variance trade-off.

Five Strategies

1. Regularisation

Add a penalty term to the loss function that discourages large weights. This constrains the model's capacity without changing its architecture. Covered in detail in Section 8. Common types: L1 (Lasso), L2 (Ridge), Elastic Net (combination).

2. More Training Data

With more data, the training set becomes a better representation of the true distribution, reducing the chance of memorising noise. This is the most reliable remedy, but is often expensive or impossible. There is a point of diminishing returns — beyond a certain dataset size, adding more data yields negligible improvement.

Learning Curve Intuition As n → ∞: Training Error ↑ (harder to memorise more data)
As n → ∞: Test Error ↓ (better estimate of true distribution)
As n → ∞: Gap between train and test → 0 (generalisation improves)

3. Simpler Model

Reduce model complexity directly. Examples: use fewer features, reduce tree depth, use fewer hidden layers in a neural network, decrease polynomial degree. This moves the model toward the underfitting side of the bias-variance trade-off.

4. Dropout (Neural Networks)

During training, randomly "turn off" (set to zero) a fraction of neurons at each layer. This prevents neurons from co-adapting and forces the network to learn redundant, robust representations. At test time, all neurons are active but their outputs are scaled down. Dropout is equivalent to training an ensemble of exponentially many sub-networks and averaging their predictions.

Dropout Mechanism Training: rⱼ ~ Bernoulli(1 − p) (each neuron kept with probability 1−p)
ŷ = r ⊙ y (element-wise multiply — zeroed neurons contribute nothing)
Testing: ŷ = (1 − p) × y (scaled to maintain expected output)

5. Early Stopping

Monitor validation loss during training. Stop training when validation loss stops decreasing (or starts increasing), even if training loss is still falling. This prevents the model from over-training on the training set. Most practical for iterative algorithms like gradient descent.

Analogy — Studying for an Exam

  • Regularisation: Setting a rule — "I will only study the top 3 most important topics deeply" — constrains what you focus on.
  • More data: Getting additional past papers from previous years — more examples to study from.
  • Simpler model: Choosing to study a concise summary guide instead of the entire textbook.
  • Dropout: Occasionally closing your eyes and trying to recall a topic from memory — forces you to build robust understanding rather than relying on one specific cue.
  • Early stopping: Recognising when further study is not improving your mock exam scores and stopping — more cramming won't help and may cause burnout.

Key Takeaway

Preventing overfitting requires a multi-pronged approach. Regularisation and early stopping are the most universally applicable. Dropout is specific to neural networks. More data is the most reliable but often impractical. Simpler models trade some training performance for better generalisation. The best strategy often combines several of these techniques.

Feature Engineering Inside the Pipeline

Feature selection and engineering are powerful ways to reduce model complexity and prevent overfitting. However, they must be done inside the cross-validation pipeline — otherwise, information from the validation set leaks into training.

Definition — Data Leakage via Feature Selection

If you select features using the entire dataset before splitting for CV, the feature selection process has seen the validation data. The selected features may be optimal for the validation set, producing an overly optimistic CV score. Feature selection must happen only on the training fold within each CV iteration.

Feature Importance Methods

Method How It Works Best For
Gini / Impurity Importance Sum of impurity reductions at splits using a feature, normalised Tree-based models (Decision Trees, Random Forests, XGBoost)
Permutation Importance Shuffle one feature at a time; measure the drop in model performance Any model; model-agnostic; more reliable than impurity importance
Correlation / VIF Correlation matrix or Variance Inflation Factor to detect multicollinearity Linear models; identifying redundant features

Correct vs Incorrect Pipeline

✓ Correct Pipeline

Correct: Feature Selection Inside CV
1 For each CV fold:
a) Split into train fold and validation fold
b) Select top features using only the train fold
c) Train model on selected features from train fold
d) Evaluate on validation fold using same selected features
→ No information leakage

✗ Incorrect Pipeline

Incorrect: Feature Selection Before CV
1 Before any CV:
a) Select top features using entire dataset
b) Then run CV on the selected features
→ Validation folds influenced the selection
→ CV score is optimistically biased
→ Real performance will be worse

Data Leakage Checklist

  • Feature selection done before CV? Leakage!
  • Scaling/normalisation done before CV? Leakage! (Use fit_transform on train, transform on validation)
  • Imputation done before CV? Leakage! (Learn imputation statistics from train only)
  • Feature engineering (e.g., target encoding) done before CV? Leakage!
  • Any preprocessing that uses validation data statistics? Leakage!

Key Takeaway

All preprocessing — feature selection, scaling, imputation, encoding — must happen inside the CV loop, fitted only on the training fold. Using sklearn's Pipeline class makes this automatic and foolproof. Never fit any transformer on the full dataset before cross-validation.

Common Misconceptions

Several persistent myths can lead practitioners astray. Here we address the most common misconceptions about model evaluation and generalisation.

✗ Misconception 1: "High R² means a good model" ▼

Reality: R² measures how much variance in the target variable is explained by the model — on the training data. A model with 100 features on 50 samples can achieve R² = 0.99 while being completely useless on new data. R² on training data tells you nothing about generalisation.

Correct approach: Always report R² on the test set (or via cross-validation). Compare it to training R² — a large gap signals overfitting. Also check if R² is meaningfully above a simple baseline (e.g., predicting the mean).

Key Point

A high training R² with a low test R² is the textbook signature of overfitting. R² is only meaningful when measured on unseen data.

✗ Misconception 2: "More features always improve the model" ▼

Reality: Adding irrelevant features introduces noise and increases variance without reducing bias. This is the curse of dimensionality — in high-dimensional spaces, data becomes sparse and models struggle to find meaningful patterns. Adding features also increases computational cost and the risk of overfitting.

Correct approach: Use feature selection (L1 regularisation, permutation importance, correlation analysis) to keep only informative features. A model with 10 well-chosen features often outperforms one with 100 noisy features.

Key Point

Quality trumps quantity in feature engineering. Irrelevant features increase variance, slow training, and can actually decrease test performance.

✗ Misconception 3: "Cross-validation trains the final model" ▼

Reality: Cross-validation is an evaluation procedure, not a training procedure. The models trained during CV folds are discarded — they exist only to estimate generalisation performance. The final model must be retrained on all available training data (train + validation) after CV is complete and hyperparameters are selected.

The workflow:

  1. Use CV on the training set to evaluate different configurations
  2. Select the best hyperparameters based on CV scores
  3. Retrain the final model on ALL training data with the selected hyperparameters
  4. Evaluate once on the held-out test set

Key Point

CV evaluates; it does not produce the final model. After selecting hyperparameters via CV, retrain on all training data before the final test evaluation.

✗ Misconception 4: "Normalising/standardising always helps" ▼

Reality: Feature scaling is essential for distance-based algorithms (KNN, SVM, K-Means) and gradient-based algorithms (neural networks, logistic regression). However, tree-based models (Decision Trees, Random Forests, Gradient Boosting) are invariant to feature scaling because they make splits based on individual feature thresholds, not distances or gradients. Scaling does not help them and adds unnecessary computation.

Correct approach:

  • Scale: Linear models, SVM, KNN, Neural Networks, PCA
  • Don't bother: Decision Trees, Random Forests, XGBoost, LightGBM
  • Always inside CV: When you do scale, use fit_transform() on training folds and transform() on validation folds

Key Point

Scaling is algorithm-dependent. Tree-based methods don't need it. When scaling is needed, always fit the scaler on training data only to prevent leakage.

Exam Tip

These misconceptions are favourite exam traps. If a question states "the model achieved 99% R² on training data," always ask: "What is the R² on the test set?" If a question says "we added 50 features and training accuracy improved," ask: "Did test accuracy improve?"

Flashcards

What is the decomposition of expected test error?

E[Error] = Bias² + Variance + Irreducible Noise (σ²)

Bias² captures systematic error from model simplification. Variance captures sensitivity to training data fluctuations. Noise is inherent randomness that no model can eliminate.

What is the key difference between the validation set and the test set?

The validation set is used for model selection and hyperparameter tuning. The test set is held out until the very end and is used only once for the final, unbiased performance estimate. Using the test set for decisions invalidates its role as an honest measure of generalisation.

Why is Leave-One-Out (LOO) cross-validation sometimes problematic?

High computational cost (N model fits) and high variance in the estimate. Each training set differs by only one sample, so the K estimates are highly correlated. The overlap between training sets means the variance of the average can actually be higher than K-Fold CV with K=5 or K=10.

What is the difference between L1 and L2 regularisation?

L2 (Ridge) shrinks all weights toward zero but never makes them exactly zero. L1 (Lasso) can set weights to exactly zero, performing automatic feature selection. L2 is preferred when all features are relevant; L1 when many features may be irrelevant.

What is data leakage and why is it dangerous?

Data leakage occurs when information from outside the training set influences the model. It produces overly optimistic performance estimates that fail in production. Common causes: feature selection before CV, scaling before split, target leakage, temporal leakage.

What is the purpose of nested cross-validation?

The outer loop provides an unbiased estimate of generalisation performance. The inner loop performs hyperparameter selection. Without nesting, the hyperparameter tuning process itself overfits to the validation data, producing an optimistically biased performance estimate.

What are the five main strategies for preventing overfitting?

1. Regularisation (L1/L2 penalty). 2. More training data. 3. Simpler model (fewer features, reduced complexity). 4. Dropout (randomly zero neurons during training). 5. Early stopping (halt training when validation loss stops improving).

Why must feature selection happen inside the CV loop?

To prevent data leakage. If features are selected using the full dataset (including validation folds), the selected features may be optimised for the validation data, producing an overly optimistic CV score. Each fold's feature selection must use only that fold's training data.

What is an overfit model's characteristic error pattern?

Training error → near zero; test error → very high. The gap between training and test error is large. The model has memorised noise and specific patterns of the training data rather than learning the general signal.

Which CV strategy should you use for time-series data and why?

Time-Series Split. Standard K-Fold randomly assigns samples to folds, which can put future data in the training set and past data in the validation set — causing temporal leakage. Time-Series Split ensures the training set always precedes the validation set chronologically.

What happens to model performance as λ (regularisation strength) increases?

Bias increases, variance decreases. At λ=0: no regularisation (potential overfit). At optimal λ: balanced bias-variance. As λ→∞: all weights → 0, model predicts the mean always (severe underfit). The optimal λ is found via cross-validation.

Does feature scaling help tree-based models like Random Forest?

No. Tree-based models make splits based on individual feature thresholds, not distances or gradients. Scaling does not change the relative ordering of threshold-based splits. Scaling is useful for distance-based (KNN, SVM) and gradient-based (neural networks, logistic regression) methods.

Scenario Question

Scenario: Building a Disease Prediction Model

A hospital wants to build a model to predict whether patients will develop a specific disease within 5 years. They have 2,000 patient records with 47 features each (age, blood pressure, cholesterol levels, genetic markers, lifestyle factors, etc.). The dataset is moderately imbalanced — 15% positive cases (diseased), 85% negative.

The data science team follows these steps:

  1. They compute correlation of each feature with the target, select the top 15 features using the entire dataset
  2. They standardise all features using the full dataset statistics
  3. They train a Logistic Regression model on 80% of the data (1,600 samples) and test on 20% (400 samples)
  4. The model achieves 92% accuracy on the test set. The team celebrates and plans deployment

Questions:

  1. Identify all data leakage issues in the team's pipeline.
  2. Why is 92% accuracy potentially misleading given the class imbalance?
  3. Describe the correct pipeline the team should follow instead.
  4. What cross-validation strategy would you recommend and why?
  5. How should the team handle the class imbalance in their model?
Show Solution ▼

1. Data Leakage Issues

  • Feature selection before train/test split: Computing correlations with the target using the full dataset means information from the test set has influenced which features are selected. The 20% test set has "leaked" into the feature selection process.
  • Standardisation before split: Computing mean and standard deviation from the full dataset means the test set statistics have influenced the scaling. The test set should be completely invisible until final evaluation.

2. Why 92% Accuracy is Misleading

With 85% negative cases, a model that predicts "no disease" for every patient achieves 85% accuracy. The team's model only improves on this baseline by 7 percentage points. More importantly, accuracy does not distinguish between catching actual diseases (recall) and false alarms (precision). If the model has low recall, it is missing sick patients — which is the whole point of the system. The team should report precision, recall, F1 score, and AUC-ROC.

3. Correct Pipeline

  1. Split data into training (80%) and test (20%) sets once, stratified to preserve class balance
  2. All preprocessing (feature selection, scaling, imputation) must happen inside the CV loop on training folds only
  3. Use sklearn's Pipeline to chain preprocessing and model
  4. Run Stratified K-Fold CV on the training set for model selection and hyperparameter tuning
  5. Select best model, retrain on ALL training data
  6. Evaluate once on the held-out test set

4. Recommended CV Strategy

Stratified K-Fold CV (K=5 or K=10) because the dataset is moderately imbalanced (15% positive). Stratification ensures each fold preserves the 15/85 class ratio, preventing folds that happen to have too few or too many positive cases. Standard K-Fold might create folds with very few positive examples, producing unreliable estimates.

5. Handling Class Imbalance

  • class_weight='balanced' in Logistic Regression — automatically adjusts weights inversely proportional to class frequencies
  • Use appropriate metrics: F1 score, AUC-ROC, or precision-recall AUC instead of accuracy
  • Threshold tuning: The default 0.5 threshold may not be optimal for imbalanced data; adjust based on the precision-recall trade-off
  • Consider SMOTE or other resampling techniques inside the CV pipeline (not before splitting)

Final Recap

Key Concepts Summary

  • Generalisation is the ability to perform well on unseen data. Overfitting occurs when a model memorises training noise instead of learning the true signal — producing near-zero training error but very high test error.
  • The Three-Way Split separates data into training (model fitting), validation (model selection / hyperparameter tuning), and test (final unbiased evaluation). The test set must NEVER be used for any decisions.
  • Bias-Variance Decomposition: E[Error] = Bias² + Variance + σ². Increasing model complexity reduces bias but increases variance. The optimal model sits at the minimum of the U-shaped total error curve.
  • K-Fold Cross-Validation rotates which data serves as validation, giving every sample a chance to be both training and validation. K=5 or K=10 are standard choices; LOO for very small datasets.
  • CV Strategies: Standard K-Fold (general), Stratified K-Fold (imbalanced classification), Time-Series Split (temporal data), Group K-Fold (grouped/correlated data), LOO (tiny datasets).
  • Hyperparameter Optimisation: Grid Search (exhaustive), Random Search (faster, often comparable), Bayesian (uses past evaluations to guide search). Nested CV separates tuning from evaluation to avoid optimistic bias.
  • ERM Re-Anchor: K-Fold CV solves the overfitting problem; regularisation constrains the hypothesis space; hyperparameter optimisation selects the hypothesis class.
  • Regularisation: L2 (Ridge) shrinks all weights; L1 (Lasso) drives weights to exactly zero for feature selection. λ controls the bias-variance trade-off and is tuned via CV.
  • Preventing Overfitting: Regularisation, more data, simpler models, dropout, and early stopping — often used in combination.
  • Feature Engineering: All preprocessing must happen inside the CV pipeline to prevent data leakage. Use sklearn's Pipeline for automatic, correct application.
  • Common Misconceptions: High training R² ≠ good model; more features ≠ always better; CV evaluates, it doesn't train; scaling doesn't help tree-based models.

Big Picture Takeaway

Model validation and generalisation form the bridge between building a model and deploying it reliably. The bias-variance trade-off provides the theoretical framework; cross-validation, regularisation, and hyperparameter optimisation provide the practical tools. The single most important principle is this: never evaluate your model on data that has influenced any of your decisions — whether those decisions involve feature selection, hyperparameter tuning, or model architecture. Keeping this discipline gives you honest, deployable models.
